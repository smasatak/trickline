# スノーボード動画分析アプリ「trickline」要求ドキュメント

- バージョン: v0.3（2026-07-14）
- v0.1からの主な変更: MVPを「自分の動画2本の比較」に再定義し、お手本動画ライブラリをフェーズ1.5に後ろ倒し。まずはスマホ単体で自分・知人に渡してPOC検証する方針を反映。
- v0.2からの主な変更: `docs/feature_requests.md`の要望を整理し反映。フェーズ1.5に「撮影直後の即比較」(F-14)を追加、骨格推定の可視化フィジビリティスパイク（D-08）を実施、3D視点変換・お手本ライブラリの投稿主体をオープンイシュー化。

---

## 1. プロダクト概要

### 1.1 目的
スノーボーダーが**オフトレ施設（サマーゲレンデ・ブラシ斜面・エアマット等）**で自分の技の動画を撮影し、比較・分析することで上達を支援するアプリ。個人の記録・分析にフォーカスする。雪山ゲレンデでの利用は対象外とする（D-13）。

### 1.2 対象ユーザー
- ライト〜初心者：基礎姿勢を確認したい層
- 中級者：自己分析をして弱点を把握したい層
- 超上級者・オフトレ層：技の反復練習を高頻度で行い、細かい差分を確認したい層

### 1.3 差別化ポイント
2本の動画を横並び・重ね合わせで比較できる点。将来的には骨格推定を用いた自動位置合わせにより、単なる動画比較ツールを超えたフォーム診断ツールを目指す。

## 2. スコープ

### 2.0 POC方針

まずはApp Store / Google Play展開や本格的なユーザー管理を前提にせず、スマホ単体で自分・知人に渡して使える範囲で検証する。

POCで優先すること:

- スマホで撮った動画2本をその場で比較できるか。
- 基準点を合わせて、スロー/コマ送り/同時再生が練習に使えるか。
- キッカー/ジブのトリック比較で、成功/失敗の違いを見つけやすいか。

POCでは後回し:

- App Store / Google Play公開準備。
- ストア審査、課金、ネイティブ配布設計。
- 大規模ユーザー管理、公開プロフィール、共有コミュニティ。
- 本格的なクラウド保存/配信基盤。

### 2.1 MVP（フェーズ1）に含める機能
- 動画読み込み（スマホ端末内の自分の滑走動画）
- 横並び再生（**自分の動画2本**の同時再生・再生速度調整・手動タイミング合わせ）
- スロー再生・ステップ送り
- 最小限の動画管理（端末内/ローカルでの選択、技タグ、メモ）

> MVP検証の仮説：「2本の動画を同期比較できること」自体に需要があるか。お手本動画がなくても、昨日の自分／今日の自分、成功トライ／失敗トライの比較で検証可能。

### 2.2 フェーズ1.5（MVP検証後、需要が確認できたら）
- お手本動画ライブラリ（運営側管理画面、タグ付け・カテゴリ分類、検索・絞り込み）
- お手本動画のH.264への手動トランスコード（クロスデバイス再生互換のため）
- お手本動画の提供元との利用許諾整理
- ~~タイマー付き撮影→即比較（F-14）~~ → POCに前倒しで導入済み（D-14、3.5節参照）

### 2.3 フェーズ2以降
- 骨格推定（MediaPipe Pose等）による自動位置合わせ
  - **着工条件**: 実際のオフトレ施設映像（ダボついたウェア・モーションブラー・ブラシ斜面背景）でMediaPipeの推定精度を確認するスパイクを先に実施し、実現性を評価する → **D-08で実施済み・クリア**
- フェーズ2の初期対象は**キッカー・ジブのトリック**に絞る
  - 基準点は基本的に「抜け」で一本化する。一連の流れなので、抜けが揃えば全体の比較軸も揃う前提で扱う
  - POC初版UIではキッカー/ジブ選択は必須にしない。将来の保存タグ、分析チェックポイント、骨格推定の分類として扱う
  - 将来論点: 抜け/オーリーの瞬間をどこまで自動判定できるか。自動候補が出せる場合は、ユーザーが「合わせる」ボタンで反映できるようにする
  - キッカー/ジブの進入、空中姿勢、板角度、重心、着地は、基準点候補ではなく比較・分析時のチェックポイントとして扱う
- 重ね合わせ表示（オーバーレイ）
- 関節角度・重心位置の数値化
- 過去の自分の動画との時系列比較
- 技の成功率トラッキング
- コーチ・スクール向けの注釈機能（動画への描き込み）

## 3. 機能要件（MVP）

### 3.1 動画アップロード
| ID | 要件 |
|---|---|
| F-01 | ユーザーは自分の滑走動画をアップロードできる |
| F-02 | 対応フォーマット：mp4, mov（スマホ標準フォーマット） |
| F-03 | アップロード時に技名・カテゴリ（グラトリ／ジャンプ／基礎姿勢等）をタグ付けできる |
| F-04 | サムネイルは**アップロード前にクライアント側で生成**する（video要素＋canvasでフレームを静止画化し、動画と一緒にアップロード）。サーバー側のffmpeg／ワーカー基盤は持たない |
| F-05 | 動画は**1本あたり最大60秒**。アップロード前にトリミングを促すUIを設ける（コスト管理と、比較したい瞬間にすぐ到達できるUXの両方のため） |
| F-06 | 動画本体はpresigned URLでCloudflare R2へ直接アップロードする。APIサーバーは動画バイナリを扱わない |

注記: F-01〜F-06のクラウドアップロード要件は、POC段階では必須ではない。まずは端末内動画/ローカル読み込みで比較体験を検証し、展開フェーズでクラウド保存・配信・アカウント連携を扱う。

### 3.2 比較再生
| ID | 要件 |
|---|---|
| F-20 | ユーザーは自分の動画から2本を選択し、横並びで同時再生できる |
| F-21 | 再生速度を0.25倍〜1倍の範囲で変更できる |
| F-22 | 各動画の開始位置（タイミング）を手動で合わせられる（例：離陸の瞬間をタップしてマーク）。オフセットはComparisonSessionとして保存する |
| F-23 | 約1/30秒単位のステップ送り・ステップ戻しができる（HTML5 videoにフレーム厳密なAPIはないため近似で可。`requestVideoFrameCallback`の利用を検討、Safari対応に注意） |
| F-24 | シークバーで両動画を同期して操作できる |

### 3.3 動画管理
| ID | 要件 |
|---|---|
| F-30 | ユーザーは自分がアップロードした動画の一覧を閲覧できる |
| F-31 | ユーザーは自分の動画を削除できる |
| F-32 | 動画に技名・日付・メモを付与できる |

### 3.4 お手本動画ライブラリ（フェーズ1.5に後ろ倒し）
| ID | 要件 |
|---|---|
| F-10 | 運営側がお手本動画を登録・管理できる管理画面がある |
| F-11 | お手本動画は技名・難易度でタグ付けされ、検索・絞り込みができる |
| F-12 | ユーザーは自分のレベル・練習したい技に応じてお手本動画を選択できる |
| F-13 | お手本動画は登録時にH.264へトランスコードする（iPhone撮影のHEVC動画がAndroid等で再生できない問題への対策。ユーザー自身の動画は自身の端末で見るため対象外とする割り切り） |

### 3.5 タイマー撮影→即比較（POCに前倒しで導入、D-14）
| ID | 要件 |
|---|---|
| F-14 | ユーザーはアプリ内のセルフタイマーカメラで「何秒後に撮影を開始するか」「何秒間撮るか」を設定して撮影できる。撮影した動画はアップロードを経由せず、そのまま比較スロット（A/B）に投入して即比較できる |

### 3.6 骨格推定の可視化（POCに前倒しで導入）
| ID | 要件 |
|---|---|
| F-40 | 比較再生中の各動画に対し、滑走者の骨格（MediaPipe Pose）をオーバーレイ表示できる。表示はON/OFF切替可能 |
| F-41 | 推論はブラウザ内（MediaPipe Tasks for Web / WASM）で行い、バックエンドを必要としない（POC方針D-09と整合） |
| F-42 | 検出信頼度が低いフレームでは骨格を描画しない（誤った骨格を出すより非表示を優先する） |

> フェーズ2の「自動位置合わせ」は引き続きフェーズ2のまま。前倒しするのは可視化のみ。

## 4. 非機能要件

### 4.1 インフラ・コスト
| 項目 | 方針 |
|---|---|
| ストレージ | Cloudflare R2（Egress無料。10GBまで無料枠のため検証期間はストレージコストほぼゼロ） |
| 動画配信・エンコード | MVPでは自前トランスコードを持たず、アップロードされた動画をそのまま配信。フェーズ1.5でお手本動画のみH.264化。品質課題が顕在化したらCloudflare Streamを検討 |
| バックエンド | FastAPI + PostgreSQL（asyncpg）。既存の技術スタックを流用 |
| 想定規模 | 個人開発規模（〜数千ユーザー）を前提としたコスト設計 |

注記: POC段階ではバックエンド/クラウドストレージは必須ではない。スマホ単体またはローカル配布で比較体験の検証を優先する。

### 4.2 パフォーマンス
- 動画はR2へ直接アップロードするため、APIサーバーの負荷・帯域は動画サイズに依存しない
- アップロード中もアプリの他機能を利用可能にする

### 4.3 セキュリティ・法務
- ユーザーがアップロードする動画は個人の滑走動画のため著作権侵害リスクは低いが、第三者の映り込みを考慮し利用規約に明記する
- お手本動画の提供元（プロライダー・施設等）との利用許諾は**フェーズ1.5（お手本ライブラリ導入時）に整理する**。MVP段階では対象コンテンツが存在しないため対応不要
- 個人情報保護方針を定める（アカウント情報、位置情報を扱う場合）

### 4.4 対応プラットフォーム
- **モバイルWeb（レスポンシブ）を優先**。後にPWA化を検討
  - 理由：ストア審査なしで配布でき検証が速い／`<input accept="video/*">`でカメラロールからのアップロードが可能／2動画同期再生はHTML5で実装可能
  - 注意点：iOS Safariでは`playsinline`必須。フレーム厳密なコマ送りは不可（F-23参照）
- App Store / Google Play展開は将来の展開フェーズで**実施する意向**（POC段階のスコープには入れない）。方式はCapacitor等でのラップを本線とし、やることは [`docs/appstore-expansion.md`](appstore-expansion.md) にまとめてある
- ネイティブアプリ（フルネイティブ書き直し）は、Webでは実現できない体験が必須になった段階で検討する。

## 5. データモデル（概略）

### 5.1 主要エンティティ（MVP）
- **User**：ユーザー情報（アカウント、レベル自己申告）
- **UserVideo**：ユーザーがアップロードした動画（R2オブジェクトキー、サムネイルキー、技タグ、アップロード日時、メモ、動画長）
- **ComparisonSession**：比較再生の設定（比較した2本の動画、各動画のタイミングオフセット）

### 5.2 フェーズ1.5で追加
- **ReferenceVideo**：お手本動画（ファイルパス、技名、難易度、提供元情報）
  - ComparisonSessionは「UserVideo × UserVideo」に加えて「UserVideo × ReferenceVideo」の組み合わせを許容するよう拡張

### 5.3 フェーズ2で追加予定
- **PoseKeypoints**：骨格推定結果（フレームごとの関節座標）
- **TrickAttempt**：技のトライ記録（成功・失敗、日時）

## 7. オープンイシュー（要決定事項）

1. **収益化方針**：個人課金（サブスク等）かB2B（施設・スクール向け）か。B2Bに進む場合はマルチテナント化（コーチが生徒の動画を閲覧する権限モデル）がデータモデルに影響するため、**フェーズ2の設計時点までに方向を決める**
2. **動画の保存期間・容量上限（アカウント単位）**：1本60秒制限は決定済みだが、アカウントあたりの総容量・本数上限は未定。無料枠を超えるユーザーが現れた時点で決定で可
3. **お手本動画ライブラリへの投稿主体**：運営のみが登録するのか、ユーザーも投稿できるのかが未定。誰でも投稿可とする場合は権利処理・モデレーションの設計が追加で必要になるため、フェーズ1.5着手前に決める
4. **3D視点変換（自由視点化）の要否**：単眼動画からの3D化は工数・難易度が大きいため、フェーズ2（2D骨格推定）のフィジビリティスパイク結果が出た後に、着手するかどうかを再検討する
